# Cross-Disease Densification Queue

这页不是再开新 disease。

这页只回答一个问题：

`五病种现在既然都已经进入 output-capable wiki surface，下一轮跨病种 densification 最该补哪里？`

## One-Line Summary / 一句话总结

现在最该对齐的，不再是 `有没有 output`，而是：

- full-text / image-table / official-source precision
- bilingual family breadth
- regulatory thickness
- control-layer maturity

而且在当前分工下：

`front-door and presentation shaping are no longer the content-side default task`

## Current Cross-Disease Read / 当前跨病种读法

### CKD

**Current position**
- mature template
- strongest control layer
- bilingual and output layers are already broad
- regulatory densification now also has a comparison layer between archetype strength and route cleanliness

**Best next densification**
- selective regulatory or route refinement only, preferably through comparison-layer clarification rather than more route-memo proliferation
- avoid low-signal page sprawl

### IBD

**Current position**
- deepest extraction density and now 24/24 explicit full source-card depth
- strongest bilingual reuse coverage
- regulatory branch still thinner than the rest of the module
- regulatory densification has now started to stabilize into `product archetype` plus `claim-fit / route-fit boundary`

**Best next densification**
- product-type-first regulatory densification beneath the new claim-fit / route-fit boundary layer
- full-text / image-table precision only where it sharpens workup sequence, treatment claim-fit, or route-fit
- control-layer refinement only where it improves reuse

### HCM

**Current position**
- output and bilingual layers are now real
- compiled spine is coherent
- source-card evidence density is now complete at 24/24 explicit full; remaining gaps are full-text/output-specific precision and absent regulatory corpus
- endpoint/workup densification has now started to stabilize into `frontier augmentation` and `biomarker use-case` rather than only broad endpoint wording

**Best next densification**
- treat seed-scope extraction and source-card thickening as complete; move into full-text/output-specific precision only where structural pressure is real
- sharpen treatment hierarchy and biomarker use-case separation only when a concrete output needs stronger claim ranking
- keep regulatory branch thin unless real corpus support arrives

### FIP

**Current position**
- output and bilingual core stack now exist
- disease-form, mechanism, and treatment branches are already strong
- source-card evidence density is now complete at 24/24 explicit full
- main gap is now control thickness, full-text/official-source precision, image/table verification, and family breadth, not front-door absence or generic source-card thickening
- endpoint/workup densification has now started to stabilize into `support order` and `assay boundary` rather than only broad topic wording

**Best next densification**
- treat seed-scope ingest, round-1 extraction, and source-card thickening as complete, then densify only where control or reuse pressure is still thin
- deepen endpoint hierarchy and diagnostic-workup architecture beneath the new support-order and assay-boundary layer
- expand bilingual family only where reuse pressure is real
- keep regulatory branch as a stop-sign page until official-source ingest exists

### Diabetes

**Current position**
- fifth 24-source module is now present
- 24/24 source cards and 24/24 round-1 worksheets exist
- first topic family and thirteen narrow owner memos exist
- U.S. SGLT2 regulatory / current-label control exists
- first briefing / dossier / slides output set now exists across `working-en / en / zh`
- first high-reuse bilingual compiled dashboard now exists
- main gap is full-text clinical maturity and non-U.S. regulatory status, not bootstrap coverage

**Best next densification**
- deepen remission, diet/obesity sequencing, insulin protocol, and SGLT2 safety only through full-text or regulatory precision where output demand requires it
- add non-U.S. regulatory checks only if jurisdiction comparison becomes necessary
- add more bilingual topic pages only if the new dashboard proves real cross-language reuse
- avoid product-ranking or treatment-leaderboard language until comparator/full-text depth supports it

## Default Cross-Disease Priority Order / 默认跨病种优先顺序

1. after seed-scope `24 / 24` completion and 24/24 full source-card depth, deepen thin-but-important full-text/image/regulatory layers before adding more page families
2. expand bilingual coverage only after one-language reuse pressure is already clear
3. densify regulatory only when product/archetype or official-source structure is real
4. prefer control-layer thickening over new memo proliferation once output families already exist
5. when a recurring gap is structurally narrow, prefer a `narrow owner` over another broad summary page

## Immediate Highest-Value Queue / 眼下最高价值队列

### Tier A

- IBD: full-text/image/regulatory precision for diagnostic-boundary, workup-sequence, and treatment/claim-fit owners
- FIP: endpoint / workup precision and control-layer thickening, now via narrower owners rather than generic source-card thickening or more broad memo shells

Post-full-source-card promotion batches should use:

- [content precision promotion workflow](content-precision-promotion-workflow.md)

### Tier B

- HCM: full-text/output-specific precision only; no generic source-card thickening remains
- CKD: only narrow refinement, especially comparison-layer refinement rather than new branch sprawl
- Diabetes: selective full-text or regulatory deepening only where output demand or jurisdiction comparison makes it necessary

既然普通用户入口和前端展现已经交给另一侧处理，当前这边默认不再优先做：

- front-door simplification
- display-layer cleanup
- presentation-first navigation shaping

- [content-side densification queue](content-side-densification-queue.md)

`truth-layer densification and content-state write-back`

## What Not To Do Next / 下一步不该做什么

- do not translate broadly just to make bilingual coverage look symmetrical
- do not promote thin regulatory branches into route recommendations
- do not add more memo shells where the real gap is evidence density or control alignment
