## One-Line Summary / 一句话总结

既然前门和展现层已经被分出去，内容侧下一轮默认不该再做：

- front-door simplification
- presentation polish
- navigation display shaping

内容侧现在最该做的是：

`full-text / official-source / image-table precision plus content-state write-back`

## Default Content Rule / 内容侧默认规则

优先补：

- `order`
- `boundary`
- `use-case`
- `comparison`

不要优先补：

- 新 front-door page
- presentation-layer simplification
- 只为“看起来完整”而加的 family page

如果当前问题重复出现的不是某个疾病内容边界，而是同一类内容工作又被重新要求一次，先按：

- [durable agent codification protocol](durable-agent-codification-protocol.md)

判断它是否已经应该从一次性执行变成唯一 system owner。

如果同类工作已经明确是 full-source-card 之后的内容 promotion 判断，直接走：

- [content precision promotion workflow](content-precision-promotion-workflow.md)

## What Counts As A Good Content Move / 什么算高质量内容动作

一个高质量内容动作，通常同时满足：

1. it sharpens a recurring structural judgment
2. it can be written as a narrow reusable owner
3. it changes how dashboards, translation pages, or synthesis pages should now be read

## What Not To Spend Content Cycles On / 内容侧现在不该花力气的地方

- 不要继续把普通用户前门当主线任务
- 不要把 README 继续改成产品落地页
- 不要把 navigation 当成主要内容 densification 容器
- 不要为了和前端保持对称而提前改 presentation-facing wording

## Write-Back Rule / 默认回写规则

每做完一个高价值窄 owner，默认同轮回写：

1. one-language topic page
2. dashboard
3. bilingual high-reuse page, if reuse is already real
4. cross-disease owner, if the default read of the module changes
