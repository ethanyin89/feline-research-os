这篇文章开头的例子非常强：一个 agent 花两小时做市场研究，另一个 agent 第二天接着写战略备忘录，人类回来审查“当时到底决定了什么、为什么这么决定”。但问题是，研究在一个聊天窗口里，备忘录在另一个窗口里，中间的推理链条没有地方保存。文章直接把这定义为当下 agent 工作“无法被继续构建”的根因。

它的核心不是 memory，也不是 orchestration，而是 preservation layer，保存层。文章说得很清楚：早期 CommonGround 更偏 orchestration，想象的是 swarm-like execution 和 multi-agent collaboration；但这个 preview 把重心收缩了，不再试图拥有所有 workflow、runtime 或 memory system，而是保存那些让人、agent、工具、运行时可以协调的“public facts of work”。

这句话非常重要。它实际上把 agent 系统拆成了三类东西：

第一类是执行层：谁去做任务，哪个 agent 调哪个工具。

第二类是记忆层：以后怎么总结、压缩、检索、复用经验。

第三类是公共事实层：这项工作当时被请求了什么、交接了什么、返回了什么、交付了什么、未来可以依赖什么。

CommonGround Kernel 选择做第三层。它不做完整 agent runtime，也不做自动编排框架，也不声称已经是完整 memory system。文章明确说它是 kernel，不是 hosted product；是 memory-ready，不是 memory-complete；也不是 agent runtime 或 automatic orchestration framework。

这和文章的词汇高度一致：kernel、foundation、common ground、durable records、public facts、commons。这些词本身就带有建筑感、地基感、公共空间感。配图如果做成普通 AI 机器人或聊天气泡，就会很廉价；但如果做成类似“空间、结构、层、节点、轨迹、容器、光线”的抽象画面，就会高级很多。

你可以把它的视觉方法总结成一句话：

用“建筑化的抽象空间”来表现“agent 工作的公共记忆”。

这对你做 Feline Research OS 的视觉非常有启发。你不一定要画猫、实验室、论文、医生。更高级的方向是：

用“研究地形图”表现猫疾病知识；  
用“分层档案库”表现文献证据；  
用“发光路径”表现研究推理链；  
用“透明容器/标本盒”表现可审查的实验记录；  
用“网格化生态系统”表现疾病模型、药效、PK、临床方案之间的连接。

也就是说，你的产品视觉不应该只是“可爱猫 + 医疗科技”，而应该是“猫研究的知识基础设施”。

我觉得你可以从这篇文章里抽取两个资产。

一个是产品内核文案：

“No feline research works alone.”  
任何猫病研究、药物评价、文献检索、方案设计，都不应该孤立在一次聊天里。它应该变成可保存、可复核、可交接、可复用的研究记录。

另一个是视觉方向：

Feline Research OS 不是宠物内容工具，而是一个 research commons。画面可以做成温暖的米白/灰蓝底色，中央是猫研究对象或疾病模型，周围是文献卡片、实验读数、时间线、样本、数据库节点，形成一个柔和但高度秩序化的研究空间。

所以，这个附件对你最重要的启发不是“CommonGround 这个项目能不能直接用”，而是它给了你一个很高级的产品判断：

未来的 agent 产品，不是谁生成得更快，而是谁能把工作留下来，变成别人和未来自己可以继续构建的地面。  
  
你这个提醒是对的。这里先把边界收住：我只围绕 `feline-research-os` 项目来读这份附件，不再延伸到爱宠客产业快讯、媒体内容生产、行业日报那套逻辑。

这份 `psql_bm25s` 的文章，对你的 feline-research-os 最重要的启发不是“要不要马上用 Postgres BM25 扩展”，而是一个更底层的判断：

你的产品如果只是一个会回答猫研究问题的 Streamlit 页面，它很容易停留在 demo；但如果它能把每次检索、每次方案生成、每次证据筛选、每次人工修正都沉淀成可低成本反复检索的项目状态，它才会变成真正的 Research OS。

文章开头的判断非常关键：长时间运行的 agent 真正输掉的不是 context window，而是 retrieval。随着任务跑到第 20 个 session、第 3 周、第 100 次工具调用，真正的问题变成：agent 能不能便宜地重新读取它已经积累过的旧工具输出、昨天的纠正、上周的引用、以及一直在处理的应用数据。文章还指出，这类累计状态在生产系统里通常已经和应用一起存在 PostgreSQL 里。

这句话放到 feline-research-os 里，可以翻译成：

你的系统最重要的能力，不是一次性塞进多少 feline disease 文献，也不是一次 prompt 写得多长，而是以后每次研究猫 FIP、CKD、HCM、FCGS、PK/PD、药效评价时，系统能不能快速找回以前已经做过的证据、判断、排除理由和方案版本。

也就是说，feline-research-os 的核心矛盾不是“模型不够聪明”，而是“研究状态不能低成本复用”。

这篇文章第二个重要点是 retrieval as a budget item。它说，在 agent 系统里，检索是一个预算项；如果 search call 很慢，harness 就会 ration retrieval，把检索从默认动作变成 fallback，agent 得到的记忆切片会变窄，证据 grounding 也会更晚发生。反过来，一旦检索成本低到可以忽略，agent 就可以在每一步运行多个 focused queries，随时重新检查累积状态，更早发现自己的错误。

这对你的产品特别重要。因为 feline-research-os 未来不应该是：

用户问一个问题 → LLM 查一次 → 输出一个答案。

而应该是：

用户问一个问题 → 系统先查内部研究记录 → 再查权威文献 → 再查标准/指南/实验模板 → 再把已有结论和新证据对齐 → 最后输出。

比如用户问：“猫 CKD 药效评价方案里，应该选哪些 biomarker？”

一个弱系统会直接生成：

SDMA、creatinine、BUN、UPC、phosphorus……

一个 Research OS 应该先检索：

以前是否做过 CKD biomarker 任务；
以前是否保存过 IRIS 分期依据；
以前是否写过猫 CKD 模型说明；
以前是否有人修正过 SDMA 的适用边界；
以前是否生成过方案模板；
当前 PubMed 有没有新的 feline CKD biomarker 文献；
这些证据分别适合诊断、入组、疗效终点还是安全性观察。

所以，你要关注的不是“有没有检索”，而是“检索是否廉价到足以成为每一步的默认动作”。

第三个关键点，是这篇文章对 psql_bm25s 的定位非常克制。它明确说 psql_bm25s 不是 agent runtime，不是 reranker，也不是 memory policy engine，而是一个小的、精确的 retrieval primitive。它负责让上层系统可以便宜、可靠地查找信息；至于存什么、什么时候总结、什么时候遗忘，是上层应用自己的决策。

这也非常适合你现在的阶段。feline-research-os 目前不需要一上来做复杂 agent runtime，也不需要做庞大多 agent 编排。你最应该先做的是一个小而清晰的 retrieval primitive：

能保存研究记录；
能保存文献卡片；
能保存疾病模型卡片；
能保存实验指标卡片；
能保存方案版本；
能按关键词、疾病、物种、证据类型、任务类型快速找回。

从这个角度讲，你的最小架构可以是：

Streamlit 前端负责交互；
SQLite 或 Postgres 存储结构化记录；
BM25 / full-text search 做低成本关键词检索；
可选向量检索做语义补充；
LLM 只在检索结果之后进行综合。

也就是说，LLM 不是中心，retrieval record 才是中心。

第四个关键点，是文章非常强调 structured retrieval，而不是 search box。它说 agent state 很少只存在一个 text column 里；一个 memory record 会有 statement、source、timestamp、confidence label、task context；应用记录也有 names、statuses、owners。真正的 grounding 不是找到一段文本，而是找到“带足够上下文的正确文本”。

这句话对 feline-research-os 几乎可以直接落地。你的每条知识不应该只是一段 markdown，而应该是结构化记录。

比如一条 feline CKD biomarker 记录，最好不是这样：

“SDMA can be used as an early biomarker of feline CKD……”

而应该是：

疾病：feline CKD
物种：cat
证据类型：guideline / review / original research / internal protocol
指标：SDMA
用途：early detection / staging support / monitoring
适用阶段：screening / enrollment / endpoint / safety observation
来源：PubMed / IRIS / internal note
可信度：high / medium / low
更新时间：2026-xx-xx
备注：不能单独替代完整肾功能评估
关联任务：CKD 药效评价方案、CKD biomarker 文献综述

这才是 Research OS，而不是资料库。

第五个关键点，是 base-plus-delta maintenance。文章说真实应用数据不是 append-only：记录会被编辑，agent memory 会被纠正，工具输出会随着项目成熟被改写成更耐用的形式。它们提出 base-plus-delta 模型：base 是优化后的 BM25 index，delta 记录最近的写入、更新和删除；后台再周期性合并。

你不一定马上实现这个工程结构，但这个思想非常适合 feline-research-os 的知识管理。

因为你的研究知识一定不是静态的。比如：

一开始，FIP 模型记录可能只是粗略描述；
后来你补充了 qPCR、SAA、A/G、病理、IHC；
再后来你根据客户反馈调整了入组标准；
再后来你发现某个表述不够严谨，需要修正；
再后来你把它沉淀成标准方案模板。

所以 feline-research-os 里的知识最好分成两类：

一类是 stable base：长期稳定的基础知识、疾病框架、模型标准、指南共识、成熟方案模板。

一类是 active delta：最近新增文献、临时判断、某次任务的输出、用户修正、未完全验证的想法。

系统回答时，应该同时看 stable base 和 active delta，但在展示时要区分“成熟结论”和“近期新增/待验证记录”。

这会比普通 RAG 高一个层级。

第六个关键点，是 harness loop。文章说他们构建 psql_bm25s 的方式不是相信 agent 的单次输出，而是让每个优化都经过同一个循环：提出 hot path 假设、实现修改、跑前后 benchmark、在所有约束下比较、决定是否保留。没有任何优化可以只因为“听起来合理”就留下。

这对 feline-research-os 的启发是：你也需要一个 Research Harness，而不是只做 prompt。

比如你每次让系统生成“猫疾病模型方案”，不应该只看它写得像不像，而要有固定验证面：

是否说明疾病定义；
是否区分猫和犬；
是否列出模型类型；
是否说明入组标准；
是否列出主要读数；
是否区分诊断指标、疗效指标、安全性指标；
是否引用来源；
是否标注证据等级；
是否暴露不确定性；
是否给出下一步可执行动作。

这就是 feline-research-os 的 harness。

从这篇文章反推，你的项目最好不要先追求“越来越智能”，而是先追求“每一次输出都能被验证、保存、检索、复用”。

我建议你把 feline-research-os 的实现逻辑调整成下面这个内核：

第一，Research Record 是主对象。每次用户任务都生成一条研究记录，而不是只生成聊天结果。

第二，Evidence Card 是基本知识单元。每篇文献、每个指南、每个实验指标、每个模型经验，都拆成可检索卡片。

第三，Retrieval 默认发生。每次回答前，系统都应该先检索已有记录和知识卡片，而不是让 LLM 裸答。

第四，Output 必须带结构。输出不只是自然语言，而是包含 request、scope、evidence、decision、answer、uncertainty、next step。

第五，人工修正要入库。你每次觉得“这个表述不严谨”“这个指标不能这么说”“这里犬猫不能混用”，都应该变成新的 correction record。

第六，稳定知识和临时知识分层。成熟方案模板、共识、指南进 stable base；新文献、新想法、未验证判断进 active delta。

如果极简落地，可以先不碰 psql_bm25s。因为它是 Postgres extension，偏工程基础设施，不一定适合你现在的第一步。你可以先做一个轻量版本：

Streamlit + SQLite/Postgres + markdown/json records + Python BM25/FTS + LLM synthesis。

等数据量、任务量和复用频率真的上来，再考虑 Postgres-native BM25 或类似实现。

但它给你的方向是明确的：feline-research-os 不是一个“猫研究问答网页”，而应该是一个“猫研究工作记忆系统”。

你现在要抓的不是“更会生成”，而是：

以前做过什么，能不能找回来；  
为什么当时这么判断，能不能看见；  
哪些证据被使用，能不能复核；  
哪些修正发生过，能不能继承；  
同类任务再来一次，能不能站在上一次基础上继续。  
  
这份附件对 `feline-research-os` 的价值非常明确：它补上的不是“知识库怎么建”，而是“这个系统如何长期工作而不半途自以为完成”。

如果前面两份材料分别对应：

CommonGround Kernel：研究工作要留下可恢复、可复用、可交接的公共记录。
psql_bm25s：这些记录要能被低成本、结构化、反复检索。
那么这份 RALPH → Zenith 补的是第三块：
`feline-research-os` 需要一个 harness，也就是控制循环，来决定每一轮研究任务如何推进、如何检查、何时继续、何时停止。

这份文章的核心判断是：长任务失败，常常不是因为模型不会做，而是因为它过早宣布完成。文章把这种现象称为 premature completion：agent 做出一个看起来合理的结果，运行了它自己选择的检查，写出自信总结，然后在关键要求还没满足时停止。

这对 `feline-research-os` 非常关键。因为你的系统面对的不是“一问一答”，而是猫疾病研究、文献筛选、实验设计、方案生成、证据复核这类长链条任务。一个普通 LLM 很容易输出一份看起来很完整的“猫 FIP 药效评价方案”，但里面可能漏掉入组标准、读数时间点、阴性/阳性对照、伦理要求、统计学考虑、证据等级，或者把犬猫数据混用了。问题不是它完全不会写，而是它会在“看起来像完成”时停下来。

文章把长程 agent 能力明确拆成模型能力和 control loop。模型决定单次 session 里能做什么；harness 决定 session 如何排列、状态如何保存、进度如何检查、什么时候继续或停止。文章列出的核心控制机制包括 state preservation、gap-finding、revisable planning、independent verification、stopping discipline。

放到 `feline-research-os` 里，这五个机制可以直接翻译成产品内核。

State preservation：每一次研究任务都必须留下记录。比如用户问过什么、检索了哪些文献、选了哪些证据、生成了什么方案、用户后来修改了哪里。否则下一次同类任务又从零开始。

Gap-finding：每一轮输出后，不是问“写完了吗”，而是问“相对原始研究目标还缺什么”。比如生成猫 CKD biomarker 方案后，要反查是否覆盖了诊断、分期、入组、疗效终点、安全性观察、随访指标、证据来源。

Revisable planning：不要一开始定死完整流程。猫疾病研究常常查着查着会发现问题改变了。比如一开始是“找 CKD biomarker”，后来发现真正要解决的是“哪几个 biomarker 适合作为药效评价终点”。系统计划必须能随证据变化而调整。

Independent verification：不能让生成方案的同一个 worker 自己宣布合格。需要一个独立检查器，从证据、物种、终点、方案完整性、术语严谨性几个角度审查。

Stopping discipline：不要无限迭代，也不要过早停止。系统需要判断“继续查是否还可能显著改善结果”。比如已经找到指南、综述、关键原始研究，且方案检查通过，就可以停止；如果缺少猫种属原始研究，就应该提示还不能关闭任务。

这份文章里 RALPH 的逻辑尤其适合作为你当前版本的第一步。RALPH 很简单：每个新 session 都重新打开原始需求和当前项目状态，然后只问一句：当前状态和用户需求之间还有什么 gap，并关闭它。文章说，这是治疗 premature completion 最简单的办法，但代价是每一轮都要重新发现 gap，而且缺少原则性的停止规则。

`feline-research-os` 可以先不做完整 Zenith，但可以先做一个 Feline-RALPH 循环：

第一轮：生成初稿。
第二轮：读取原始任务 + 初稿，找缺口。
第三轮：补缺口。
第四轮：独立检查是否满足任务。
第五轮：生成最终版本与 research record。

比如用户输入：

“帮我设计一个猫 FIP 药效评价方案。”

普通输出是直接写方案。
Feline-RALPH 输出应该是：

先生成方案草案；
再检查是否缺疾病定义、模型来源、入组标准、分组、给药、监测时点、SAA/qPCR/A-G/HCT 等读数、出组标准、统计学、伦理、人道终点、局限性；
再补齐；
再由 verifier 检查是否存在犬猫混用、证据缺失、过度承诺、注册表述不严谨；
最后才输出。

这就从“会写”变成了“会完成”。

但文章也提醒，RALPH 虽然强，但成本高，因为它反复重新找 gap，而且没有清晰停止规则。Zenith 的价值在于保留 repeated gap-finding，同时让每一轮更有针对性：动态分配 worker、tester、reusable skills、replanning 和 stopping decisions。文章强调，Zenith 不是简单多跑几轮，而是根据当前状态决定下一步需要什么类型的工作、验证或策略变化。

这对你未来版本很重要。`feline-research-os` 不应该永远只有一个“Research Agent”。它可以逐步拆成几个角色，但这些角色不是为了炫技，而是为了控制质量。

我建议你把未来的 `feline-research-os` harness 设计成五类内部角色：

Research Worker：负责检索和初步整理文献、指南、内部记录。

Protocol Worker：负责把证据转成方案结构，包括动物、分组、终点、采样、统计、伦理。

Evidence Verifier：专门检查来源是否可靠、证据是否支持结论、是否有 PubMed/指南/内部记录支撑。

Species & Veterinary Verifier：专门检查犬猫是否混用、猫种属特殊性是否被忽略、兽医术语是否准确。

Final Synthesizer：只在前面检查通过后，负责输出最终版本和 research record。

这就是把 Zenith 的 adaptive orchestration 轻量化移植到你的项目里。

文章中还有一个非常重要的判断：Plan-RALPH 说明“规划”本身不是万能的。固定的 upfront plan 会过期，因为计划是在执行者真正学习到项目约束之前写出来的；而且如果“done” 标签由同一个 worker 自己标记，就会变成假完成。

这对你的产品是一个提醒：不要把 `feline-research-os` 设计成一套过度固定的流程按钮，比如“第一步查文献、第二步总结、第三步写方案、第四步输出”。这可以作为默认路径，但不能成为死流程。因为不同任务的真实缺口不同。

比如：

FIP 药效方案的缺口可能在模型稳定性和读数体系。
CKD 方案的缺口可能在分期和长期随访。
HCM 方案的缺口可能在影像学终点和自然病程。
PK 方案的缺口可能在采血时间点、样本量、分析方法和动物福利。
FCGS 方案的缺口可能在病例异质性和评分标准。

所以更好的方式是：先有默认模板，但每轮都允许系统重新判断“当前最缺的是证据、方案结构、验证，还是表达修正”。

这份文章最适合转化成 `feline-research-os` 的一个内部标准：不要只做 Prompt，而要做 Harness。

我建议你把项目内部架构写成这样：

`Feline Research OS = Knowledge Base + Retrieval Layer + Research Record + Harness Loop + LLM Workers`

其中这份 RALPH/Zenith 文章主要对应 Harness Loop。

一个最小可落地版本可以是：

用户提交任务后，系统创建一条 `research_record`。
系统运行 `initial_answer`。
系统运行 `gap_check`，对照原始任务和输出找缺口。
如果缺口明显，运行 `revision`。
系统运行 `independent_verification`。
如果 verifier 失败，回到 revision。
如果 verifier 通过，输出 final answer，并保存 sources、decisions、uncertainties、next steps。

你现在用 Streamlit 做，也完全可以实现这个逻辑，不需要复杂工程。一开始可以把它做成页面上的几个状态：

Draft generated
Gap review completed
Revision completed
Verification completed
Final research record saved

用户看到的不是“AI 一次性回答”，而是一个研究任务被推进、被检查、被封存的过程。

这份文章还特别强调 stopping discipline：继续工作是否值得成本。 对 `feline-research-os` 来说，停止规则可以很实用：

当以下条件满足时，任务可以进入 final：

原始问题已被逐项回应；
关键证据来源已列出；
物种范围明确，没有犬猫混用；
输出中区分了确定结论和不确定判断；
方案类任务包含目的、动物、分组、指标、时间点、统计、伦理、风险；
文献类任务包含筛选范围、文献卡片、Quick take、下一步研究分支；
verifier 没有发现高风险问题。

当以下情况出现时，不应该 final：

缺少猫种属证据；
只引用综述，没有原始研究或指南；
关键实验终点没有来源；
模型或疾病定义不清；
结论超出证据；
用户目标发生变化但计划没有更新。

这就是 feline-research-os 从 demo 走向专业工具的分界线。

最后，我觉得这份附件给你的产品定位补了一句很关键的话：

`feline-research-os` 的竞争力，不是“模型更聪明”，而是“控制循环更可靠”。

别人也可以接 PubMed，也可以调用 DeepSeek 或 GPT，也可以做 Streamlit 页面。但如果你的系统能做到：每次都保存状态、每次都找 gap、每次都独立验证、每次都知道何时继续和停止，那么它就不只是一个猫研究聊天工具，而是一个真正能长期承担研究任务的 OS。  
  
这份 `II-Thought` 对 `feline-research-os` 的补充点，和前面三份材料不一样。

前面三份分别在讲：

CommonGround Kernel：研究工作要留下 durable record。
psql_bm25s：这些 record 要能被低成本检索。
RALPH / Zenith：长任务要有 harness，防止过早完成。

而 `II-Thought` 这一篇补的是第四块：
`feline-research-os` 里的“问题、答案、证据、验证标准”本身要被高质量地构造出来。

也就是说，它不是在讲 agent 怎么跑，而是在讲：什么样的数据，才值得进入一个 reasoning system。

这篇文章开头就把 II-Thought-RL-v0 定位为一个大规模、多领域的强化学习数据集，目标是提供高质量、大规模的 reasoning question-answer pairs，并为后续更复杂的 reasoning traces 打基础。它提出的四个核心原则是 Scale、Quality、Verifiability、Integrity，其中 Verifiability 指每个样本要支持自动评估和强化学习应用，Integrity 指严格去重和去污染，避免和专有 benchmark 重叠。

这对 `feline-research-os` 的启发非常直接：你的系统不能只是把猫疾病资料不断堆进去，而要把资料转化成高质量、可验证、可复用的“研究任务样本”。

对于 feline-research-os 来说，最小的数据单元不应该只是“文献摘要”，而应该逐步变成：

一个明确问题；
一个可核查答案；
一组支持证据；
一个适用边界；
一个 verifier 或 checklist；
一个任务类型标签。

比如，不是简单存：

“猫 CKD 可以用 SDMA、肌酐、尿素氮等指标评价。”

而是存成一个更像 II-Thought 样本的结构：

问题：在猫 CKD 药效评价方案中，SDMA 更适合作为入组筛查指标、疗效终点，还是安全性观察指标？
答案：SDMA 可用于肾功能评估和早期肾损伤提示，但不能单独替代完整 CKD 分期与疗效评价体系；在方案中更适合作为辅助肾功能指标，与 creatinine、BUN、phosphorus、UPC、尿比重、血压等共同使用。
证据：IRIS/指南/原始研究/综述/内部模型记录。
边界：仅限猫；不把犬数据直接外推；不把诊断指标直接等同于疗效终点。
验证：回答是否区分诊断、入组、疗效终点和安全性？是否标注物种？是否引用证据？是否避免单指标过度承诺？

这就是把“知识库”升级成“可验证研究任务库”。

这篇文章第二个关键点是数据质量。它非常系统地列出低质量数据的问题：问题歧义、语言或格式问题、无关内容、重复和污染、浅层题目、题干包含答案、多部分问题、依赖外部视觉元素、不可解问题、开放式问题等。

这些问题放到 feline-research-os 里，几乎一一对应。

例如，低质量输入可能是：

“猫肾病怎么做？”
这个问题太宽，系统很容易生成泛泛答案。

更好的任务应该被重写成：

“针对自然发病猫 CKD 药效评价研究，如何设计入组标准、主要/次要终点和随访指标？”

再比如，多部分问题：

“帮我查猫 CKD 文献，顺便写方案，再做客户介绍页。”
这会让系统难以验证。

更好的方式是拆成三个任务：

猫 CKD biomarker 文献检索；
猫 CKD 药效评价方案设计；
客户沟通版服务说明生成。

再比如，开放式问题：

“猫 FIP 研究有什么价值？”
这个可以回答，但不适合作为可验证任务。

更适合 Research OS 的问题是：

“猫 FIP 药效评价中，SAA、RT-qPCR、A/G ratio 和病理/IHC 分别承担什么评价作用？”

所以，II-Thought 给你的启发是：`feline-research-os` 不能完全接受用户原始问题，而应该先做 task reformulation，把模糊问题改写成可验证任务。

这可以成为产品里的一个独立步骤：

Raw question → Research task reformulation → Retrieval → Answer → Verification → Record。

第三个关键点是“verifiable reward”。文章说 DeepSeek R1 的成功很大程度上归因于高质量、可验证的问题对；verifiable 指回答可以被确定性确认或系统性验证，从而形成客观、精确的强化学习依据。文章还提到，数学和代码是天然可验证的，医学等复杂领域可能需要 LLM judge。

这对 feline-research-os 很重要，因为你的领域不是纯数学，也不是代码，很多问题属于“半可验证”。

比如：

“猫 FIP 药效方案是否合理？”
不能用一个唯一数值答案验证。

但可以用 LLM-as-Judge + 专业 checklist 验证：

是否限定物种为猫；
是否说明 FIP 模型来源；
是否列出入组标准；
是否包含病毒学指标；
是否包含炎症指标；
是否包含血液/生化指标；
是否包含临床评分；
是否包含病理或组织学终点；
是否说明人道终点；
是否避免把模型数据说成临床疗效；
是否给出证据来源。

这就是 feline-research-os 的“半可验证任务”体系。

我建议你把系统中的任务分成三类：

第一类，完全可验证任务。
例如文献字段提取：PMID、DOI、年份、物种、疾病、研究类型、样本量。这类可以用规则和结构化校验。

第二类，半可验证任务。
例如方案是否完整、证据是否支持结论、犬猫是否混用。这类适合 checklist + LLM judge。

第三类，开放式综合任务。
例如“这个研究方向是否值得做”。这类不能直接打分，但可以要求输出 evidence、assumption、uncertainty、next step，再由人审。

`feline-research-os` 最好优先建设前两类，因为它们更容易沉淀成系统能力。

第四个关键点是两阶段质量过滤。II-Thought 的数据处理流程中，先用 Gemini 2.0 Flash 做初筛，快速识别措辞不清、信息不完整、结构有问题的问题；再用 Qwen-2.5-32B-Instruct 做更严格的质量保证，确保问题结构清晰、可验证。最后还有规则过滤，比如长度限制、排除 proof-based 问题、过滤不可解析答案、排除没有 test cases 的代码题。

这个思路可以直接迁移到 feline-research-os 的知识入库流程。

你以后把一批猫疾病资料、文献、内部方案导入系统时，不应该直接全部入库，而应该有一个 Feline Data Curation Pipeline：

第一步，去重和去污染。
同一篇文献、同一个方案版本、同一段重复模型介绍，不要反复进入系统。否则后面检索时会放大重复结论。

第二步，初筛。
剔除明显低质量内容：来源不明、表述含混、物种不清、疾病名不清、没有引用、没有上下文的片段。

第三步，专业质量审核。
判断这条记录是否清楚说明物种、疾病、研究类型、证据用途、限制条件。

第四步，结构化。
把自然语言资料拆成 Evidence Card、Protocol Card、Biomarker Card、Model Card、Verifier Rule。

第五步，任务化。
把一些内容进一步转成可验证 QA，例如“这个指标适合什么用途”“这个模型适合回答什么研究问题”“这个方案缺哪些要素”。

第五个关键点是文章对浅层样本的警惕。它特别提到 True/False、Multiple-choice、Yes/No 这类结构可能让模型 reward hacking，不是真正理解。

这对你非常有用。`feline-research-os` 里不要把 verifier 设计得太浅，比如只问：

“是否包含 SAA？是/否。”
“是否包含 qPCR？是/否。”

这会让系统变成勾选清单，但不一定真的合理。

更好的 verifier 应该问：

“SAA 在该方案中承担什么评价作用？是炎症活动度、入组筛查、疗效监测，还是安全性观察？它是否和病毒载量、临床评分、血液生化结果一起解释？”

也就是说，验证不是看关键词有没有出现，而是看这个指标是否被放在正确的研究语境里。

第六个关键点是 decontamination。II-Thought 很重视数据集和 benchmark 的重叠问题，认为 benchmark contamination 会损害模型表现的可信度。文章中甚至列出了多个数据集和 MATH500、AIME、Olympiad 等 benchmark 的重叠情况。

在 feline-research-os 里，这可以转化为另一个问题：不要让系统把“待评估材料”和“标准答案/验证依据”混在一起。

比如你以后可能会用系统生成方案，然后又让系统评价这个方案。如果评价依据来自它刚刚自己生成的内容，就会变成自我循环。

更严谨的做法是分层：

Source layer：外部文献、指南、标准、内部已确认经验。
Generated layer：模型生成的草稿、摘要、方案。
Verified layer：人工或 verifier 确认过的结论。
Benchmark/checklist layer：专门用于评价输出质量的规则，不与生成材料混用。

这样可以避免系统“用自己生成的东西证明自己是对的”。

第七个关键点是 II-Thought 特别强调 underrepresented domains：科学、工程、金融、医学等领域的数据不足，未来要扩展到这些更真实复杂的领域。

这恰好说明你的 feline-research-os 是有价值的，因为猫兽医研究就是一个典型的 underrepresented domain。大模型通用能力很强，但在“猫疾病模型—药效评价—PK/PD—临床前方案—证据边界”这种细分领域，公开可用的高质量、结构化、可验证数据很少。

所以你不是在做一个普通垂直知识库，而是在做一个稀缺领域的 reasoning dataset / research task system。

这也是为什么我觉得你不应该只停留在 Streamlit 问答。你可以慢慢把系统运行过程中产生的东西沉淀成一个内部数据资产：

Feline Research QA pairs
Feline Evidence Cards
Feline Protocol Checklists
Feline Model Evaluation Rules
Feline Biomarker Usage Map
Feline Study Design Verifiers

这类资产的长期价值，可能比当前 app 本身还大。

把这份材料合并到前面的三份材料后，`feline-research-os` 的内核可以进一步升级成：

CommonGround 负责“工作记录”：每次研究任务留下什么。
psql_bm25s 负责“可检索性”：这些记录以后能不能被低成本找回。
Zenith 负责“控制循环”：长任务如何防止过早完成。
II-Thought 负责“数据质量”：进入系统的问题、答案和证据是否高质量、可验证、去重、无污染。

所以，`feline-research-os` 的下一步不是简单加功能，而是建立四个内部标准：

第一，Research Record Standard。
每次研究任务都保存 request、scope、sources、decision、answer、uncertainty、next step。

第二，Retrieval Standard。
每次回答前默认检索历史记录、证据卡片和指南/文献来源。

第三，Harness Standard。
每个复杂任务至少经过 draft、gap check、revision、independent verification、final。

第四，Data Quality Standard。
每条入库知识都要经过去重、质量过滤、结构化、可验证性判断。

如果只抽取这篇 `II-Thought` 对你的一个最重要启发，我会这样表述：

`feline-research-os` 不应该只积累“内容”，而应该积累“可验证的研究问题与答案”。

因为只有可验证，才可能复用；只有复用，才可能形成 OS；否则它只是一个更漂亮的聊天界面。  
  
这份 `II-Researcher` 对 `feline-research-os` 的补充非常关键。前面几份材料更多在讲“长期工作怎么保存、怎么检索、怎么控制、怎么保证数据质量”，而这份材料直接对应你的产品最表层、也是用户最能感知的一层：一个研究型 Agent 到底如何从问题出发，逐步检索、压缩、反思、再检索，最后生成带引用的结构化研究结果。

如果只围绕 `feline-research-os` 来读，它的核心启发是：

`feline-research-os` 不应该只是“猫研究知识库 + LLM 总结”，而应该是一个面向猫研究问题的 Deep Research pipeline。

也就是：

用户提出猫研究问题；
系统判断这个问题需要什么样的研究深度；
系统拆出子问题；
系统检索文献、指南、内部记录；
系统压缩和筛选证据；
系统反思还缺什么；
必要时继续检索；
最后生成可复核、带引用、结构化的研究报告或方案。

这篇文章对 Deep Research 的描述很清楚：它不是普通聊天机器人，而是会迭代浏览网页、读取多个来源、把信息编译成结构化输出的研究代理；它会规划研究策略、从多个网站和文档中收集数据，并生成带引用和推理步骤的综合报告。

放到你的产品里，这意味着 `feline-research-os` 的目标不应该是“回答猫 CKD 是什么”，而应该是“像一个猫研究助理一样，完成一个可交付的研究任务”。

比如用户问：

“帮我研究猫 FIP 药效评价中哪些指标适合作为主要终点。”

普通 ChatGPT 会直接回答一堆指标。
`feline-research-os` 应该先判断：

这是一个需要多来源证据的问题；
需要区分临床症状、病毒学、炎症、血液生化、病理终点；
需要引用猫 FIP 相关研究，而不是泛泛引用犬猫感染病；
需要说明哪些指标适合 primary endpoint，哪些更适合 secondary / exploratory endpoint；
需要把证据不足的地方标出来。

这就是 II-Researcher 给你的第一层启发：Research Agent 的第一步不是回答，而是 query evaluation。

文章中的 state-machine pipeline 第一环就是 Query Evaluation，系统会判断初始问题对 Freshness、Plurality、Completeness 的要求：是否需要最新信息，是否需要多个视角或来源，是否需要完整详细的解释。

这可以直接变成 `feline-research-os` 的问题评估器。你可以把它改成更适合猫研究的版本：

Freshness：这个问题是否需要最新文献？例如新 biomarker、新药物、新疗法、新注册要求，就需要。
Plurality：这个问题是否需要多个来源？例如方案设计、终点评价、疾病模型，通常需要文献、指南、内部模型经验共同支持。
Completeness：这个问题是否需要完整方案？例如“设计研究方案”比“解释一个指标”需要更完整结构。
Species-specificity：是否必须限定猫？如果是猫研究，必须防止犬、人、小鼠证据直接外推。
Evidence type：需要指南、综述、原始研究、病例研究、内部记录，还是注册/伦理依据。
Output type：用户要的是文献卡片、研究报告、方案、表格、还是可复制 prompt。

这样，系统就不会所有问题都走同一条流程。

第二个重要启发是“状态机式研究流程”。II-Researcher 的第一种方案是 Untrained State Machine Pipeline，步骤包括 query evaluation、搜索查询生成与执行、信息检索与压缩、自我反思与批判、状态管理、最终报告生成。

这正好可以成为 `feline-research-os` 的第一个工程实现版本。你不需要一开始做复杂多 agent，只要做一个清晰状态机：

`evaluate_query`：判断用户问题类型和研究深度。
`generate_subqueries`：生成 PubMed / PMC / 内部知识库检索问题。
`retrieve_sources`：检索外部文献和内部记录。
`compress_sources`：保留最相关证据片段。
`reflect_gaps`：判断还缺什么证据。
`followup_retrieve`：必要时继续查。
`draft_answer`：生成初稿。
`verify_answer`：检查证据、物种、术语、完整性。
`final_report`：输出最终研究结果。
`save_record`：保存 research record。

这和前面 Zenith/RALPH 的 harness 可以自然接起来。II-Researcher 负责“研究流程”，Zenith/RALPH 负责“不要过早完成”。

第三个关键启发是 Context Compression。文章说，直接把所有内容塞进 LLM 在质量和成本上都不理想，而且可能超过上下文长度；但过度压缩或遗漏必要信息也会显著降低 pipeline 表现。因此他们采用 LLM + embedding 的混合压缩方法：先分段，再用 embedding 过滤相关 chunk，再用 LLM 做 generative retrieval，识别和排序相关 chunk，最后通过投票和排名机制压缩文本。

这对 `feline-research-os` 特别现实。因为猫研究文献、指南、内部方案、PDF 表格都可能很长。如果你直接把整篇文献丢给模型，成本高、噪音大，而且模型可能抓不住关键证据。

你需要的是 Feline Context Compression：

先按段落或章节切分文献。
每个 chunk 带上标题、PMID/DOI、疾病、物种、研究类型。
用关键词/BM25/embedding 找出和当前问题最相关的片段。
再让 LLM 只选择真正有用的片段，而不是重写全文。
最终压缩成 Evidence Snippet，保留原文来源和用途标签。

比如检索猫 CKD biomarker 时，系统不需要保留整篇论文，只需要保留：

研究对象是否为猫；
样本量；
疾病阶段；
检测指标；
主要发现；
适用边界；
与当前问题的关系。

这样才能让后面的回答更稳定。

第四个关键启发是 Self-Reflection Mechanism。文章说，不能完全依赖模型自己选择正确动作，所以每次访问网站后，agent 都要评估：获得了什么新知识，之前做过什么，还缺什么信息，下一步应该怎么做。

这可以直接变成 `feline-research-os` 每次检索后的固定反思卡片：

`knowledge_gained`：这次检索确认了什么？
`evidence_quality`：证据是指南、综述、原始研究，还是弱证据？
`species_relevance`：是否直接来自猫？还是犬/人/小鼠间接证据？
`remaining_gaps`：还缺什么？比如缺原始研究、缺终点定义、缺时间点、缺样本量。
`next_action`：继续检索、生成草稿、请求用户澄清，还是进入验证。

这个机制非常重要。因为 `feline-research-os` 的价值不在于“查到了一些东西”，而在于它能知道“查到的东西够不够”。

比如系统查到三篇猫 FIP 综述后，不应该马上写方案，而应该反思：

这些是综述，不足以支撑具体实验终点；还需要原始研究或模型评价数据；如果没有，则应在最终报告中标注证据限制。

第五个关键启发是 II-Researcher 比较了两条路线：一种是结构化 State Machine；另一种是 Reasoning Model with tools。文章认为，状态机方式流程清楚，但可能过于泛化、上下文意识不足；而 DeepSeek R1 / QwQ 这类 reasoning model 会在内部动态执行动作和反思，但长上下文、多次工具使用后可能忘记初始指令，因此他们用 prefilled thinking process 给模型一个结构化脚手架。

这对你很重要，因为你之前提到 DeepSeek V4 Pro 的写作和推理体验很好，也在用 gstack / Karpathy LLM wiki 的理念。这里的建议是：

`feline-research-os` 不要二选一。
最好的方式是 State Machine + Reasoning Model hybrid。

也就是：

外层用固定状态机保证流程不乱；
内层用强 reasoning model 处理复杂判断。

例如外层固定要求：

必须先评估问题；
必须检索；
必须压缩证据；
必须反思 gap；
必须验证；
必须保存记录。

但在每一步内部，可以让 DeepSeek / GPT 这类 reasoning model 做更灵活的判断：

哪些子问题最关键；
哪些证据更可信；
哪些指标适合 primary endpoint；
哪些地方需要谨慎表述；
哪些方案风险最大。

这样既不会变成死板 pipeline，也不会变成模型自由发挥。

第六个关键启发是“prefilled thinking process”的产品化意义。虽然我们不需要在用户界面暴露模型内部推理，但你可以把这种思想转化成“预填研究脚手架”。

例如用户问任何猫研究问题时，系统内部都预填一个研究框架：

我需要先明确物种、疾病、任务类型；
我需要判断是否需要最新文献；
我需要优先检索猫种属证据；
如果猫证据不足，必须标注外推来源；
我需要区分诊断指标、入组指标、疗效终点、安全性指标；
我需要输出证据、判断、限制和下一步。

这就是 `feline-research-os` 的 domain-specific prefill scaffold。它不一定要叫 thinking process，而可以叫：

`Feline Research Reasoning Scaffold`

它会让模型在处理猫研究问题时更稳定，不会每次都从通用知识问答开始。

第七个关键启发是 II-Researcher 的 benchmark 视角。它使用 GAIA 和 FRAMES 评估研究 agent，FRAMES 关注 RAG 系统的 factuality、retrieval accuracy 和 reasoning capabilities。

对你的项目来说，不能只凭“感觉回答得不错”来判断产品变好了。你也需要一个小型 Feline Research Benchmark。

可以先做 20–50 个内部测试问题，例如：

猫 FIP 药效评价有哪些核心指标？
猫 CKD 研究中 SDMA 的用途和限制是什么？
猫 HCM 研究中超声心动图终点如何选择？
猫 PK 研究采血设计有哪些约束？
猫 FCGS 临床研究评分体系如何构建？
猫皮肤真菌模型适合评价哪些产品？
猫肥胖模型中体重、BCS、代谢指标如何组合？
猫研究方案中哪些地方最容易犬猫混用？

每个问题配一个人工认可的参考答案/checklist。以后你改 prompt、换模型、加检索，都用这套问题回归测试。这样你的产品迭代就不只是“感觉更好”，而是有评价面。

把这份 `II-Researcher` 和前面材料合在一起，现在 `feline-research-os` 的内核已经很清楚了：

CommonGround Kernel 给你 Research Record。
psql_bm25s 给你低成本 Retrieval。
RALPH / Zenith 给你 Harness Loop。
II-Thought 给你 Data Quality 和 Verifiable QA。
II-Researcher 给你 Research Pipeline：query evaluation、search、compression、reflection、state management、final report。

所以，`feline-research-os` 可以定义成：

一个面向猫研究的深度研究系统，不是单轮问答工具。它通过结构化研究流程，把猫疾病、模型、文献、指标、方案和验证规则组织成可检索、可复用、可审查的研究工作系统。

如果下一步落地，我建议你优先做一个最小闭环，而不是加很多功能：

用户输入研究问题。
系统先输出“问题评估”：这是文献检索、方案设计、指标解释，还是证据审查。
系统生成 3–5 个子问题。
系统检索内部记录和外部文献。
系统压缩成证据卡片。
系统输出“当前证据是否足够”的自我反思。
如果不足，继续一轮检索。
如果足够，生成最终报告。
最后保存 research record。

这个闭环一旦跑通，`feline-research-os` 就已经从 Streamlit 应用升级成了真正的 Research Agent。  


这份 `Common Ground` 材料，和前面那篇 `CommonGround Kernel preview` 有一个明显区别：前者更像“内核层宣言”，强调 durable work records；这份更像“产品层/工作台层说明”，强调人和 AI 如何在一个可观察、可干预、可复用的工作环境里协作。

只围绕 `feline-research-os` 来看，它补上的不是检索、不是数据质量、也不是研究 pipeline，而是一个更直接的问题：

你的 feline-research-os 应该是一个“猫研究回答工具”，还是一个“猫研究协作工作台”？

这份材料的答案很明确：应该向后者靠拢。

它开头把 Common Ground 定义为一个 open-source application and framework，目标是建立人和 AI 的真正 partnership，而不只是一个强大的 multi-agent system。它特别强调：和在黑箱后面工作的 answer engines 不同，Common Ground 让 reasoning 保持开放，用户可以实时和事后检查、引导、修正 AI 团队的每个决策。

这对 `feline-research-os` 很关键。因为猫研究、动物实验、药效评价、PK/PD、疾病模型这些任务，用户不应该只拿到一个“最终答案”。用户需要知道：

系统为什么这样拆问题；
用了哪些证据；
哪些证据是猫直接证据；
哪些是犬、人、小鼠或体外研究外推；
哪些地方系统不确定；
哪些步骤可以被人工改写；
哪些判断应该被保存为以后可复用的研究经验。

所以 `feline-research-os` 的 UI 不应该只是一个输入框 + 输出框。更好的方向是：把每一次研究任务做成一个可观察的工作流。

这份材料最重要的产品结构是“三层专业团队架构”：Partner、Principal、Associates。Partner 是面向用户的 strategist，负责收集目标、澄清范围、共同制定高层计划；Principal 是执行中枢，把计划拆成具体 Work Modules、分配任务并监控进度；Associates 是具体专家 agent，比如代码分析专家、网页研究专家，独立执行任务并返回结构化 deliverables。

这套架构可以非常自然地迁移成 `feline-research-os` 的内部角色：

Partner 层可以叫 `Feline Research Strategist`。它直接和用户对话，先确认用户真正要解决的是文献综述、疾病模型分析、药效评价方案、PK 设计、终点选择，还是证据审查。它不急着回答，而是先定义任务边界。

Principal 层可以叫 `Research Coordinator`。它把任务拆成 Work Modules，比如“检索猫种属文献”“提取 biomarker 证据”“对照内部模型记录”“生成方案草案”“进行物种/证据验证”。

Associates 层可以是一组小专家：`Literature Search Associate`、`Feline Disease Model Associate`、`Protocol Design Associate`、`Evidence Verifier`、`Species-specific Reviewer`、`Report Writer`。

这不是为了搞复杂多 agent，而是为了让复杂研究任务的每一步都可见、可控、可复核。

比如用户问：“猫 CKD 药效评价方案应该怎么设计？”

普通聊天工具直接输出一篇方案。
按照 Common Ground 的思路，`feline-research-os` 应该先显示：

Partner：确认任务目标是“药效评价方案”，物种是猫，疾病是 CKD，输出类型是研究方案。
Principal：拆成四个 Work Modules：疾病与分期依据、入组/排除标准、疗效与安全性终点、随访与统计框架。
Associates：分别检索文献、提取指标、生成草案、审查证据。
Verifier：检查是否有犬猫混用、指标用途错误、证据不足或过度承诺。
Final：输出方案，并保存完整 research record。

这样用户看到的是“研究过程”，不是一段黑箱答案。

第二个重要点，是 Common Ground 强调 declarative agent personas：所有行为都放在简单、可读的 YAML profiles 里，可以不改代码就编辑 system prompt、tool menus、decision logic、self-reflection rules。它还提到 Context Handover Protocol，信息从 Partner → Principal → Associate 通过声明式交接规则传递，以避免信息丢失或漂移。

这对你的项目非常实用。`feline-research-os` 未来最有价值的地方，可能不是写一堆复杂代码，而是沉淀一组可编辑的 feline research agent profiles。

比如：

`feline_literature_reviewer.yaml`
负责 PubMed/PMC 文献检索、文献卡片、证据等级、物种标注。

`feline_protocol_designer.yaml`
负责把证据转成实验方案结构，必须覆盖目的、动物、分组、给药、指标、时间点、统计、伦理。

`species_safety_verifier.yaml`
负责检查猫种属特殊性、犬猫混用、人药/小鼠数据外推风险。

`feline_model_expert.yaml`
负责疾病模型逻辑，包括 FIP、CKD、HCM、FCGS、皮肤病、肥胖等模型的适用范围和局限性。

`research_record_writer.yaml`
负责把每次任务保存成 request、scope、sources、decisions、uncertainties、next steps。

这和你当前的 Streamlit 产品非常匹配。你不一定一开始真的用 YAML，但产品架构上要按这个思想设计：角色行为可配置、可替换、可版本化。

第三个重要点，是 Multi-view observability。Common Ground 提供 Flow、Kanban、Timeline 三种视图：Flow view 显示 AI 的 reasoning graph，Kanban view 把进度映射到项目看板，Timeline view 按时间展示执行过程。文章说，这些视图共同把 AI 的内部模型和用户的认知对齐。

这对 `feline-research-os` 的界面设计特别有启发。你现在不需要马上做复杂前端，但可以在 Streamlit 里先做简化版三视图：

Flow View：显示本次研究任务的分解结构。比如“用户问题 → 子问题 → 证据检索 → 证据压缩 → gap check → 验证 → 最终报告”。

Kanban View：显示每个 Work Module 的状态。比如 To Do、Searching、Drafting、Needs Review、Verified、Final。

Timeline View：显示系统做过什么。比如 10:12 创建任务，10:13 检索 PubMed，10:15 生成 evidence cards，10:17 发现缺少猫原始研究，10:20 追加检索，10:25 生成最终报告。

这会让 `feline-research-os` 从“AI 输出页面”变成“研究任务操作台”。

第四个重要点，是 Persistent Procedural Context。文章说 Common Ground 会把每一次运行都保存在本地，捕捉的不只是结果，而是每个决策背后的完整 procedural context，也就是 how and why；过去工作会变成 reusable intelligence。

这和前面 CommonGround Kernel、psql_bm25s、Zenith、II-Thought 都能接上。对你的项目来说，最重要的是这句话：

不要只保存 final answer，要保存 procedural context。

比如一次猫 FIP 研究任务，最终答案可能是“推荐 SAA、qPCR、A/G、临床评分、病理/IHC 作为组合终点”。但真正有价值的是过程：

为什么 qPCR 不适合单独作为主要终点；
为什么 SAA 是炎症活动指标而不是特异性病毒指标；
为什么 A/G ratio 更适合作为疾病状态和治疗反应辅助指标；
哪些文献支持，哪些只是经验；
哪些地方证据不足；
用户是否要求更偏注册、临床前还是客户沟通。

这些 how and why，才是以后 `feline-research-os` 越用越聪明的来源。

第五个重要点，是它提出的“Thinking Environment”概念。文章说，解决 cognitive offloading 的方式不是另一个黑箱 super-agent，而是一个让 AI reasoning legible、steerable，并与人类专业知识共生的 Thinking Environment。

这句话可以直接变成 `feline-research-os` 的产品定位：

它不是替代猫研究专家，而是让猫研究专家的判断过程可视化、可复用、可扩展。

也就是说，你不是要做一个“自动写猫研究报告”的工具，而是要做一个“猫研究思考环境”。

这个定位很重要，因为你自己的专业判断会始终在 loop 里。系统负责检索、整理、提醒、验证、记录；你负责定义问题、校正边界、判断策略、决定最终表达。

综合前面所有材料，现在 `feline-research-os` 的完整内核可以这样收束：

Common Ground 这份材料：产品工作台层。强调人机协作、专业团队架构、可观察流程、可编辑角色、过程记忆。
CommonGround Kernel preview：底层记录层。强调请求、交接、返回、交付、未来可依赖的 durable work records。
psql_bm25s：检索基础层。强调长任务靠低成本检索积累状态，而不是无限依赖上下文窗口。
RALPH / Zenith：控制循环层。强调 repeated gap-finding、revisable planning、independent verification、stopping discipline。
II-Thought：数据质量层。强调问题/答案/证据要可验证、去重、去污染、高质量。
II-Researcher：研究流程层。强调 query evaluation、search、compression、reflection、state management、final report。

所以 `feline-research-os` 可以明确成六层架构：

第一层，Human-in-the-loop Research Workspace。
用户不是旁观者，而是在研究过程里不断检查、修改、确认。

第二层，Professional Team Mode。
Strategist 负责澄清目标，Coordinator 负责拆任务，Associates 负责执行，Verifier 负责审查。

第三层，Research Pipeline。
问题评估、子问题生成、检索、压缩、反思、补检索、综合、验证、最终报告。

第四层，Research Record。
保存每次任务的 request、scope、sources、decisions、outputs、uncertainties、handoff、next steps。

第五层，Retrieval Memory。
让历史文献卡片、模型卡片、方案版本、修正记录、验证规则都能被低成本找回。

第六层，Data Quality & Verification。
入库前去重、结构化、标注证据类型；输出后进行物种、证据、方案完整性验证。

如果现在要把这份材料转成你产品的下一步实现，我建议优先做三个小功能，不要做大而全。

第一，在每次任务开始时增加一个 `Research Plan` 面板。显示系统如何理解问题、准备拆成哪些 Work Modules。

第二，在每个 Work Module 后增加一个 `Review / Edit` 按钮。让你可以直接修改系统判断，比如“这个任务不需要查最新文献”“这里必须限定猫”“这里要优先用 PubMed 而不是网页”。

第三，在任务结束时自动生成一个 `Procedural Record`。不是只保存最终答案，而是保存这次任务的完整路径：问题、计划、检索、证据、缺口、修正、验证、最终输出。

这三个功能一旦做出来，`feline-research-os` 的性质就会发生变化。它不再是一个生成器，而是一个可观察、可干预、可累积的猫研究工作台。  
  
这份 `Master Plan` 和前面几篇材料的性质不一样。它不是技术架构文，也不是产品实现文，而是 Intelligent Internet 这个体系的“世界观说明”。如果只围绕 `feline-research-os` 来看，它的价值不是告诉你怎么写代码，而是帮助你理解：为什么这些工具会反复强调 open、sovereign、common ground、shared knowledge、transparent coordination。

它对你的项目最重要的启发是：`feline-research-os` 不应该只是一个“我个人用来查猫研究资料的小工具”，而可以被理解为一个小型的、垂直领域的“开放知识协作系统”。

这篇文章的核心叙事是：未来智能可能被中心化在不透明的大模型里，人类的 agency 变成订阅服务；他们想走另一条路，用 open intelligence、sovereign AI、Common Ground protocol 和 shared inheritance of knowledge 来重新组织智能协作。文章明确说，他们要构建的不是由远方公司控制的单体模型，而是用户可以指挥的一组 specialist agents，并且这些 agent 应该建立在 clean、open、auditable 的人类共识知识之上。

这句话放到 `feline-research-os` 里，可以变成一个很清晰的产品原则：

不要把猫研究能力锁死在一次性 prompt 或某个黑箱模型里，而是把它拆成可拥有、可审查、可替换、可复用的专业研究组件。

比如，你现在用的是 Streamlit，可以接 GPT、DeepSeek、Kimi 或其他模型。模型可以换，前端可以换，但 `feline-research-os` 真正要沉淀的是：

猫疾病知识结构；
猫研究任务模板；
文献证据卡片；
疾病模型经验；
方案验证规则；
研究记录与修正历史；
可复用的 specialist agents。

这和文章里“不是一个 monolithic chef，而是一组 specialist cooks you command”的比喻高度一致。 对你的项目来说，就是不要幻想一个万能“猫研究大模型”一次性解决所有问题，而是建设一组你能指挥的专业小代理：

文献检索代理；
猫疾病模型代理；
PK/PD 方案代理；
药效终点代理；
证据验证代理；
研究记录代理；
最终报告代理。

这就是 `feline-research-os` 的“小型 sovereign research team”。

第二个重要点，是文章强调 Common Ground protocol 是一个 transparent coordination layer，用于让人和 agent 在不依赖许可的情况下协调、构建和解决问题。 这对你的项目非常关键。因为 `feline-research-os` 不是单纯“知识查询”，而是一个研究任务协调系统。

一个猫研究任务往往不是一步完成的。比如“设计猫 CKD 药效评价方案”，里面至少有：

疾病定义；
研究目的；
动物入组；
模型或自然病例来源；
主要/次要终点；
采样时间点；
统计分析；
伦理和人道终点；
证据来源；
风险与局限性。

这不是一个模型生成答案的问题，而是多个工作模块之间的协调问题。因此你可以把 `feline-research-os` 的核心重新定义为：

一个面向猫研究的透明协调层。

它协调的不是大量人，而是“你 + 模型 + 文献 + 内部知识 + 验证规则 + 输出模板”。

第三个重要点，是文章强调“shared inheritance of knowledge”：agent 不应建立在混乱抓取的互联网数据上，而应建立在 clean、open、auditable 的共识知识之上。 这对 `feline-research-os` 是非常直接的原则。

你的系统不能只是“联网搜一下猫病文章”。它应该优先建立自己的可信知识底座：

PubMed / PMC 文献；
兽医指南和共识；
IRIS、AAFP/ISFM、ARRIVE、PREPARE 等标准；
内部已验证模型资料；
人工确认过的方案模板；
用户修正过的术语和边界；
明确标注来源和证据等级的卡片。

这就是 feline research 的 shared inheritance。它不是泛互联网材料，而是经过筛选、去重、结构化、可审查的专业知识资产。

第四个重要点，是这篇文章虽然很宏大，但对你当前阶段的启发反而应该非常克制：不要被“Bitcoin for Intelligence Age”“permissionless society”这些大叙事带偏。对 `feline-research-os` 来说，真正可落地的是三句话：

让知识不依赖某个单一模型。
让研究过程不变成黑箱。
让每次完成的研究都能成为下一次工作的地基。

如果压缩成一个产品方向，我建议你这样理解：

`feline-research-os` 是一个面向猫研究的 sovereign research workspace。它让用户拥有自己的猫研究知识库、自己的研究流程、自己的验证规则、自己的 agent team，而不是每次把问题丢给一个不透明模型重新生成。

这和前面几份材料可以拼成一个完整逻辑链：

`Master Plan` 给的是价值观：开放、可拥有、可审查、基于共享知识。
`Common Ground` 给的是工作台：人和 AI 在可观察环境里协作。
`CommonGround Kernel` 给的是记录层：每次工作都留下 durable work record。
`psql_bm25s` 给的是检索层：长期状态要能低成本反复读取。
`RALPH / Zenith` 给的是控制层：长任务要反复找 gap、独立验证、防止过早完成。
`II-Thought` 给的是数据层：问题和答案要高质量、可验证、去重、去污染。
`II-Researcher` 给的是流程层：query evaluation、search、compression、reflection、final report。

所以，现在 `feline-research-os` 的定位可以进一步收束为：

一个以猫研究为垂直场景的开放式研究操作系统。它不是依赖单一模型的问答工具，而是把专业知识、检索、验证、研究记录和人类判断组织在一起，让每一次猫疾病/药效/PK/方案研究都可复核、可保存、可复用、可继续构建。

对下一步落地，我建议从这份 `Master Plan` 里只提取三条产品原则，不要扩散：

第一，模型可替换。
不要把系统绑定在某个模型上。GPT、DeepSeek、Kimi、Claude 都只是推理/生成层。真正要保存的是 records、cards、templates、verifiers。

第二，知识可审查。
所有重要结论都要能追溯到来源，至少区分“外部文献”“指南/共识”“内部经验”“模型生成”“人工确认”。

第三，工作可继承。
每次任务结束后，都要保存成 research record。下一次同类任务，不应该重新开始，而应该从历史记录、已验证证据和已有模板继续。

这份材料看起来很宏大，但落到 `feline-research-os` 上，最实用的一句话其实是：

不要做一个更聪明的猫研究聊天框，要做一个你自己拥有、能审查、能长期积累的猫研究智能工作台。  
  
这份 `II-Search` 对 `feline-research-os` 的补充非常具体。它不是在讲“研究工作台”这种宏观架构，而是在讲一个研究型模型如何被训练成“愿意多步搜索、会访问页面、会综合报告、会用代码化工具调用”的信息检索代理。

如果只围绕 `feline-research-os`，它补上的核心问题是：

你的系统不能只依赖一个通用大模型“愿不愿意查资料”。你需要把“多步检索行为”做成产品默认能力，甚至做成可训练、可评估、可约束的能力。

这篇文章介绍的 II-Search-4B 和 II-Search-CIR-4B，是两个 4B 参数级别的信息搜索与开放研究模型。II-Search-4B 用三阶段 SFT 加一阶段 RL，逐步刺激工具使用、多步推理和综合报告生成；II-Search-CIR-4B 则加入 code-integrated reasoning，让模型在推理过程中嵌入可执行 Python 来调用搜索和浏览 API。

这对 `feline-research-os` 的启发是：未来你不一定必须依赖最贵、最大的模型来做所有事情。很多“猫研究检索/整理/证据抽取”的动作，可以由小模型或便宜模型承担；真正复杂的专业判断，再交给强模型或人工确认。

也就是说，你的系统可以逐步形成分工：

小模型/便宜模型：负责生成检索词、筛文献、抽字段、整理 evidence cards。
强模型：负责跨证据综合、方案判断、风险边界、最终表达。
人工专家：负责最终校正和确认。

这和前面 Common Ground 的 specialist agents 逻辑是一致的。

这篇文章里最值得你关注的第一个细节，是它发现模型会出现 Tool Laziness。文章说，Phase 1 之后模型已经会调用工具，但会倾向于只搜索一次就下结论，即使问题很复杂。原因是教师模型本身知识库强，训练数据里很多问题可以 shortcut。为了对抗这个问题，他们构造了强制多轮迭代搜索的数据，最多到 14–15 turns。

这对 `feline-research-os` 很重要。因为通用模型也会有同样问题：用户问“猫 FIP 药效评价指标怎么选”，模型可能查一两篇综述就开始写。表面看完整，实际上检索深度不够。

所以你的产品里需要明确防止 “search laziness”。

可以设置不同任务的最低检索行为：

简单解释型问题：允许 0–1 次检索。
文献综述型问题：至少 3–5 个来源。
方案设计型问题：至少覆盖指南/综述/原始研究/内部记录中的 2–3 类来源。
证据审查型问题：必须检索反向证据或限制性证据。
复杂研究报告：必须多轮检索，且每轮后做 gap reflection。

这里重点不是机械规定数量，而是让系统养成一个行为：复杂问题不能一次搜索就结束。

第二个重要细节，是 II-Search 的训练分阶段很有产品映射价值。

Phase 1 是 Tool Call Ability Stimulation，先让模型学会工具调用的基本模式。
Phase 2 是 Reasoning Improvement，解决 tool laziness 和推理质量差的问题。
Phase 3 是 Comprehensive Report Generation Improvement，通过 rejection sampling 和 STORM-inspired 方法提升报告生成。
Phase 4 是 Reinforcement Learning，用奖励函数继续优化工具利用和泛化能力。

这可以变成 `feline-research-os` 的产品建设顺序：

第一阶段，不追求完美答案，先让系统稳定完成“检索—访问—抽取—引用”基本动作。

第二阶段，解决“检索太浅”和“推理太散”。也就是让系统不要查一篇就写，也不要把所有证据堆成流水账。

第三阶段，提升研究报告质量。比如输出文献卡片、Quick take、证据地图、方案建议、限制与下一步。

第四阶段，做内部评估与奖励。比如用你自己的 feline benchmark / checklist 判断答案是否合格，并把高质量输出沉淀为模板或样本。

这比“先做一个大而全系统”更稳。

第三个重点，是 II-Search-CIR 的 Code Integrated Reasoning。它让模型在特定格式的代码块里调用 `web_search` 和 `web_visit`，不仅检索外部信息，还可以在代码里处理、过滤和推理这些信息。文章认为，相比普通工具调用，这种方式更灵活、更可控。

这对 `feline-research-os` 的实际工程很有启发。你可以不用真的训练 CIR 模型，但可以把“代码化研究步骤”作为系统内部实现方式。

比如对 PubMed 文献检索，不要让模型用自然语言随意说“我查到了几篇”，而是让程序明确执行：

检索关键词；
返回 PMID 列表；
抓取 title、abstract、year、journal；
按 cat/feline、disease、biomarker、study type 过滤；
生成 evidence table；
再交给 LLM 综合。

也就是说，越是可结构化的部分，越应该用代码做；越是专业判断的部分，才交给模型。

举个例子。用户问：

“猫 CKD 中 FGF-23 是否适合作为药效评价终点？”

系统内部不应该直接让 LLM 生成答案，而应该：

用代码检索 PubMed：`feline CKD FGF-23`、`cat chronic kidney disease fibroblast growth factor 23`。
抽取每篇文章的物种、疾病、样本量、指标、结论。
过滤掉非猫或弱相关结果。
生成 evidence cards。
再让模型判断：FGF-23 更适合疾病进展/矿物质代谢相关预后指标，还是适合作为干预疗效终点？证据是否足够？

这就是 CIR 思想的产品化落地。

第四个重点，是 II-Search 的 benchmark 结果揭示了一个现象：表现更好的信息搜索模型通常会使用更多工具。比如在 Frames 上，II-Search-CIR-4B pass rate 72.2%，搜索 6.1 次、访问 5.0 次，总工具使用 11.0 次；而 Qwen3-4B pass rate 30.7%，搜索 1.1 次、访问 0.1 次，总工具使用 1.2 次。

这对你的项目有一个非常朴素但重要的启发：

研究型任务的质量，往往和“有效检索行为”高度相关。

所以 `feline-research-os` 的界面可以把工具使用透明化，让用户看到：

本次检索了几轮；
访问了多少来源；
保留了多少证据卡片；
排除了多少不相关来源；
是否进行了补充检索；
是否完成了证据验证。

这不是炫技，而是让用户判断这个研究结果是否可信。

第五个重点，是它的奖励函数非常简单但很有启发：格式错误给 -1，格式正确且答案正确给 1，格式正确但答案错误给 0。 这个思想可以迁移成 `feline-research-os` 的输出质量评分。

你的系统可以先用一个简单 scoring rubric：

格式不合格：0 分。比如没有来源、没有物种限定、没有任务结构。
格式合格但证据不足：1 分。比如写得完整，但引用不够或没有猫直接证据。
格式合格且证据支持：2 分。
格式合格、证据支持、还标注边界和不确定性：3 分。

对猫研究任务来说，格式本身不是表面问题，而是专业安全问题。没有物种、没有证据、没有边界的答案，即使语言漂亮，也不应该进入 verified record。

第六个重点，是 STORM-inspired enhancement。文章说他们借鉴 STORM，让教师模型进行互联网研究、收集参考文献，并写类似 Wikipedia 的文章，以增强模型使用 search 和 visit 工具生成长报告的能力。附录里的系统提示也很完整：先从多个 perspective 生成问题，进行模拟专家访谈式追问，再生成 outline，再写 article，最后 polish、去重、验证 claims、标注不确定性。

这对 `feline-research-os` 的“深度研究报告模式”很有价值。你可以不照搬 Wikipedia-like article，但可以借鉴它的结构：

Pre-writing Research：先从不同视角提出问题。
比如猫 FIP 主题可以从 disease biology、diagnosis、model construction、drug efficacy endpoints、animal welfare、translation limitation 六个视角提问。

Outline Generation：先生成研究报告提纲，而不是直接写正文。

Report Generation：按提纲填充证据。

Polishing：去重、修正不一致、验证 claims、标注不确定性。

这可以成为 `feline-research-os` 的一个高级模式：`Deep Research Report Mode`。

第七个重点，是 II-Search 的 prompt 强调“不要仅依赖内部知识，要使用 web search/page browsing 获取实时信息；不要在证据不足时过早 final；要用 search 做 initial discovery，用 visit 确认、深化和验证。”

这可以直接变成你的系统规则，但需要替换成 feline-research 的语言：

不要仅依赖模型内部知识回答专业猫研究问题。
优先使用 PubMed/PMC/指南/内部资料检索。
搜索用于发现，原文/摘要访问用于确认。
不能用搜索结果标题代替证据。
如果猫直接证据不足，必须标注证据层级和外推风险。
不要在未完成证据验证前输出最终方案。

这条非常重要，因为宠物药/猫疾病研究里，模型幻觉或过度外推的风险很高。

现在把这份 `II-Search` 合并到前面材料，`feline-research-os` 的技术路线可以更清楚：

Common Ground：工作台和可观察协作。
CommonGround Kernel：保存 durable research records。
psql_bm25s：让历史状态和证据可以低成本检索。
RALPH/Zenith：用 harness 防止过早完成。
II-Thought：建设高质量、可验证的研究任务数据。
II-Researcher：定义深度研究 pipeline。
II-Search：训练/设计一个愿意多步搜索、会访问、会综合、会用工具的 search agent。

所以 `feline-research-os` 现在应该有两个模式：

一个是普通 Research Assist Mode。
适合解释指标、整理单个疾病、生成简短答案。流程是轻量检索 + evidence card + answer。

另一个是 Deep Feline Research Mode。
适合文献综述、方案设计、终点选择、疾病模型评价。流程是多视角提问 + 多轮检索 + 证据压缩 + gap reflection + verifier + final report。

这两个模式要区分，不然所有问题都跑深度流程会太慢；所有问题都跑轻量流程又不可靠。

如果下一步落地，我建议你从 `II-Search` 里提取三个可执行功能：

第一，增加 `Search Depth Controller`。
根据任务类型决定检索深度。比如 quick / standard / deep 三档。Deep 模式至少进行两轮检索和一次 gap reflection。

第二，增加 `Tool Use Trace`。
在页面上显示本次用了多少次 search、visit、internal retrieval、evidence compression。用户不一定要看全部细节，但要知道系统不是裸答。

第三，增加 `Evidence-first Report Template`。
输出顺序不要一上来就是结论，而是：检索范围、证据卡片、Quick take、判断、限制、下一步。这和你前面在 II-Commons 页面截图里看到的呈现逻辑可以直接统一。

最后，用一句话概括这份材料对 `feline-research-os` 的意义：

`II-Search` 告诉你，研究型产品的关键不是“模型知道什么”，而是“模型是否被迫形成可靠的多步检索习惯”。

对于你的项目，这意味着：猫研究任务不能靠一次生成完成；系统必须默认检索、访问、压缩、反思、再验证。只有这样，`feline-research-os` 才不像一个会写答案的页面，而像一个真正的猫研究代理。
